# 프로젝트 문서화

모든 프로젝트는 아래 구조를 유지한다. 신규 프로젝트는 첫 의미 있는 코드 변경 때 함께 만든다.

```
<project>/
├── README.md         ← 한 줄 소개, 빠른 시작, 주요 entry/scope 표, docs/ 링크
└── docs/
    ├── structure.md  ← 폴더 트리, 모듈 책임, 빌드/필터 규칙, 공용 API 시그니처
    ├── apps.md       ← 실행 단위(scope/entry)별 동작 + 추가 절차 (해당 시)
    ├── hardware.md   ← 핀맵, 보드 옵션, 배선 (임베디드/하드웨어 시)
    ├── host.md       ← PC 쪽 도구/서버가 별도일 때
    └── history/      ← 날짜별 작업 내역 (`YYYY-MM-DD.md`)
```

원칙,

- 파일/라인 참조는 상대경로 markdown 링크 (`[name](path)`, `[name:42](path#L42)`).
- 코드를 그대로 옮기지 말고 WHY, 의도, 밖에서 안 보이는 제약을 적는다.
- 성격상 해당 없는 파일은 안 만든다. 새 카테고리는 `docs/` 아래 추가하고 README 링크 표에 등록.
- 깊게 안 읽은 서브트리(`host/` 등)를 문서화할 땐 그 사실을 문서 끝에 명시.

## 작업 내역 기록

- 의미 있는 코드, 설정, 문서, 규칙 변경 후 `docs/history/YYYY-MM-DD.md` 에 기록. 오늘 문서가 있으면 항목 추가, 없으면 생성.
- 항목엔 변경 파일, 이유, 검증 결과, 남은 주의점을 짧게.
- 프로젝트 `history/` 엔 그 프로젝트 루트 내부 변경만. 작업 폴더 밖(전역 규칙, 도구 설정, 다른 repo)은 직접 의존이나 사용자 영향이 있을 때만.
- 일회성 조사로 파일 변경이 없었거나 사용자가 기록 말라 하면 생략 가능.

## 업데이트 타이밍

아래 변화는 같은 PR/커밋에서 관련 문서를 함께 갱신한다 (follow-up 으로 미루지 않음).

- **폴더 구조** (디렉터리 추가/삭제/이름) → `structure.md` 트리 + README 링크.
- **실행 단위(scope, env, entry, service) 변경** → `apps.md` + README 표.
- **빌드/실행 명령** (build flag, env, 빌드 옵션, 부팅 절차) → README 빠른 시작, `structure.md` 빌드 섹션.
- **공용 모듈 public API 시그니처** → `structure.md` 모듈 섹션.
- **하드웨어 핀맵, 보드 설정, 외부 호스트/포트** → `hardware.md` 또는 `structure.md` config.
- **새 의존성(라이브러리, MCP 서버, 외부 서비스)** → 가장 가까운 문서.

불필요,

- 내부 리팩터, 버그픽스, 변수명 변경 등 외부 관찰 불가 변경.
- 일회성 실험/임시 디버깅 코드.
