# 프로젝트 문서화

모든 프로젝트는 다음 구조의 문서를 유지한다. 신규 프로젝트는 첫 의미 있는 코드 변경을 마칠 때 함께 만든다.

```
<project>/
├── README.md         ← 한 줄 소개, 빠른 시작, 주요 entry/scope 표, docs/ 링크
└── docs/
    ├── structure.md  ← 폴더 트리, 모듈 책임, 빌드/필터 규칙, 공용 API 시그니처
    ├── apps.md       ← 실행 단위(scope/entry)별 동작 + 새 단위 추가 절차 (해당될 때)
    ├── hardware.md   ← 핀맵, 보드 옵션, 배선 (임베디드/하드웨어 관련 프로젝트일 때)
    ├── host.md       ← PC 쪽 도구/서버가 별도로 있을 때
    └── history/      ← 날짜별 작업 내역 (`YYYY-MM-DD.md`)
```

원칙,

- 파일, 라인 참조는 상대경로 markdown 링크로 (`[name](path)`, `[name:42](path#L42)`).
- 문서는 코드를 그대로 옮겨 적지 않는다. WHY, 의도, 외부에서 안 보이는 제약을 적는다.
- 프로젝트 성격에 따라 위 목록 중 해당 없는 파일은 만들지 않는다. 새 카테고리가 필요하면 `docs/` 아래 추가하고 README 링크 표에 등록한다.
- `host/` 처럼 서브트리를 깊게 읽지 않은 상태로 문서를 쓸 때는 그 사실을 문서 끝에 명시한다.

## 작업 내역 기록

- 의미 있는 코드, 설정, 문서, 규칙 변경을 마치면 작업 내역을 `docs/history/YYYY-MM-DD.md`에 기록한다.
- 오늘 날짜 문서가 이미 있으면 같은 문서 아래에 새 항목을 추가한다. 없으면 `docs/history/`를 만들고 오늘 날짜 문서를 생성한다.
- 항목에는 변경 파일, 변경 이유, 검증 결과, 남은 주의점을 짧게 남긴다.
- 프로젝트의 `docs/history/`에는 해당 프로젝트 루트 내부 변경만 기록한다. 작업 폴더 바깥의 전역 규칙, 도구 설정, 다른 저장소 변경은 프로젝트에 직접적인 의존 관계나 사용자 영향이 있을 때만 기록한다.
- 일회성 조사만 하고 파일 변경이 없었거나 사용자가 명시적으로 기록하지 말라고 한 경우는 생략할 수 있다.

## 업데이트 타이밍

다음 변화가 일어났을 때 같은 PR/커밋 안에서 관련 문서를 함께 갱신한다. 별도 follow-up 으로 미루지 않는다.

- **폴더 구조가 바뀜**, 디렉터리 추가/삭제/이름 변경 → `structure.md` 트리와 README 링크.
- **실행 단위(scope, env, entry binary, service) 추가/삭제/이름 변경** → `apps.md` 와 README 표.
- **빌드, 실행 명령 변경**, build flag, env, 빌드 시스템 옵션, 부팅 절차 → README 빠른 시작, `structure.md` 빌드 섹션.
- **공용 모듈의 public API 시그니처 변경** → `structure.md` 모듈 섹션.
- **하드웨어 핀맵, 보드 설정, 외부 의존 호스트/포트 변경** → `hardware.md` 또는 `structure.md` config 섹션.
- **새 의존성(라이브러리, MCP 서버, 외부 서비스) 추가/제거** → 가장 가까운 문서.

업데이트 불필요,

- 내부 리팩터, 버그 픽스, 변수명 변경 같은 외부 관찰 불가능한 변경.
- 일회성 실험 코드, 임시 디버깅 변경.
